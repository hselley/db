Datos vs. Información

Para poder entender a fondo lo que significa diseñar una base de datos, es necesario saber la diferencia entre datos e información. Los datos son hechos en crudo, esto significa que estos hechos aún no han sido procesados para revelar su significado.

Por otro lado, la información es el resultado de procesar los datos crudos, de manera que se revela su significado.

Los datos son la base de la información que, a su vez, es el cimiento del conocimiento.

  • Los datos constituyen las piezas con las cuales se construye la información.

  • La información es obtenida al procesar los datos.

  • La información es utilizada para revelar el significado de los datos.

  • La certeza y relevancia de la información es clave para el proceso de decisión.

  • Tomar buenas decisiones es un factor clave para la supervivencia en un mundo globalizado.

Bases de Datos

Una base de datos es una estructura computacional compartida e integrada que almacena un conjunto de:

  • Datos, esto es, hechos en crudo relevantes para el usaurio.

  • Metadatos, es decir, datos acerca de los datos. A través de los metadatos los datos son administrados y almacenados.

Los metadatos proporcionan una descripción de las características de los datos y un conjunto de relaciones que enlazan los datos encontrados en la base de datos. Por ejemplo, el metadato almacena información como el nombre de cada elemento, tipo de dato (numérico, texto, etc.) para cada elemento, puede ser nulo o no, etc. Dadas las características de los metadatos, las bases de datos suelen ser conocidas como "conjunto de datos auto-descriptivos".

Sistema de Gestión de Bases de Datos

Un Sistema de Gestión de Bases de Datos (SGBD) es un conjunto de programas que administra la estructura de la base de datos y controla el acceso a los datos almacenados en la base.

El SGBD sirve como un intermediario entre el usuario y la base de datos. La estructura de la base de datos se almacena como un conjunto de archivos y la única forma de tener acceso a esos archivos es a través del SGBD. El SGBD recibe todas las solicitudes y las traduce en las operaciones requeridas para completar esas solicitudes.

Tener un SGBD entre el usuario y la base de datos ofrece diversas ventajas. En primer lugar, el SGBD permite que los datos puedan ser compartidos tanto a aplicaciones como usuarios. En segundo lugar, el SGBD las vistas de usuario de los datos en un único repositorio de datos. Las ventajas más relevantes se describen a continuación.

Ventajas de los SGBD

  • Compartición de datos mejorada. Los SGBD ayudan a crear un ambiente en el cual los usuarios tienen mejor acceso a datos mejor administrados.

  • Seguridad de los datos mejorada. Mientras más usuarios tienen acceso a los datos, mayor será el riesgo de que haya una brecha de seguridad. Un SGBD proporciona una infraestructura para una mejor privacidad de los datos y mejores políticas de seguridad.

  • Mejor integración de los datos. Amplio acceso a datos mejor administrados permite una vista clara e integrada de las operaciones en una escala mayor.

  • Inconsistencia de datos minimizada. La inconsistencia de datos ocurre cuando diferentes versiones de un mismo dato se encuentran en lugares distintos. La probabilidad de una inconsistencia de datos se reduce dramáticamente cuando se utiliza una base de datos diseñada adecuadamente.

  • Acceso mejorado a los datos. Los SGBD producen respuestas rápidas a consultas específicas. Esta es una característica implícita en el diseño e implementación de los SGBD.

  • Toma de decisiones mejorada. Datos mejor administrados y acceso mejorado a los datos, hacen posible generar información de mejor calidad, necesaria para tomar una mejor decisión.

  • Productividad del usuario mejorada. La disponibilidad de los datos, combinado con las herramientas que transforman los datos en información útil, facilitan al usuario el tomar decisiones rápidamente.

Estas sólo son algunas ventajas de utilizar un SGBD, más adelante exploraremos algunas otras.

Desventajas de los SGBD:

  • Complejidad: Los SGBD son conjuntos de programas que pueden llegar a ser complejos con una gran funcionalidad. Es preciso comprender muy bien esta funcionalidad para poder realizar un buen uso de ellos.

  • Coste del equipamiento adicional: Tanto el SGBD, como la propia base de datos, pueden hacer que sea necesario adquirir más espacio de almacenamiento. Además, para alcanzar las prestaciones deseadas, es posible que sea necesario adquirir una máquina más grande o una máquina que se dedique solamente al SGBD. Todo esto hará que la implantación de un sistema de bases de datos sea más cara.

  • Vulnerable a los fallos: El hecho de que todo esté centralizado en el SGBD hace que el sistema sea más vulnerable ante los fallos que puedan producirse. Es por ello que deben tenerse copias de seguridad (Backup).

Clasificación de las Bases de Datos

Un SGBD puede manejar muchos tipos diferentes de bases de datos. Las bases de datos pueden clasificarse por número de usuarios, ubicación de la base de datos y el tipo de uso.

El número de usuarios determina si una base de datos es mono-usuario o multi-usuario. Una base de datos es mono-usuario si sólo permite que un usuario a la vez la utilice. Por otro lado, una base de datos es multi-usuario si permite que varios usuarios la utilicen al mismo tiempo. Una base de datos multi-usuario que permite un número máximo de 50 usuarios simultáneos, se le conoce como base de datos de un grupo de trabajo. Si la base multi-usuario permite más de 50 usuarios simultáneos, se le conoce como base de datos empresarial.

La ubicación también se utiliza para clasificar las bases de datos. Si una base de datos de encuentra en un sólo lugar, se le conoce como base de datos centralizada. Si la base soporta que los datos se encuentren distribuidos en varios sitios, se le conoce como base de datos distribuida.

Una base de datos que está diseñada para almacenar las operaciones diarias de una organización, se le conoce como base de datos transaccional (también conocida como operacional o de producción). Por otro lado, una base banco de datos está enfocada en el almacenamiento de datos utilizados para generar información requerida para tomar decisiones tácticas o estratégicas.

Terminología Básica

Término Definición
Dato Los datos son hechos en crudo, esto significa que estos hechos aún no han sido procesados para revelar su significado. Los datos tienen poco significado hasta que se hayan organizado de alguna forma lógica.
Campo Un caracter o un grupo de caracteres que tienen un significado específico. Un campo se utiliza para definir y almacenar datos.
Registro Un conjunto conectado lógicamente a uno o más campos que describen una persona, lugar o cosa. Por ejemplo, un campo que constituye un registro de un cliente consiste de: nombre, dirección, número telefónico, fecha de nacimiento, etc.
Archivo Un conjunto de registros que tienen alguna relación entre si. Por ejemplo, un archivo puede contener datos acerca de los estudiantes inscritos a una Universidad.

Dependencia estructural y de datos

Un sistema de archivos posee dependencia estructural, lo que significa que para tener acceso a un archivo se depende de su estructura. Por ejemplo, si se desea agregar un campo será necesario cambiar la estructura en el archivo y modificar los datos si es necesario. Debido a este cambio, los programas que tengan acceso al archivo tendrán que modificarse para poder soportar el cambio. Por el contrario, si hay un cambio en la estructura sin afectar a los programas que tienen acceso al archivo, se dice que existe una dependencia estructural.

De manear similar, si en la estructura cambia la característica de un dato (por ejemplo un tipo de dato) y los programas que tienen acceso al archivo no soportan el cambio, se dice que existe una dependencia de datos. Por el contrario, si ante un cambio de alguna característica de los datos los programas que tienen acceso no se ven afectados, se dice que existe independencia de datos.

Redundancia de datos

La redundancia de datos ocurre cuando un dato se encuentra almacenado innecesariamente en más de un lugar a la vez.

La redundancia de datos puede ocasionar diversos problemas, tales como:

  • Baja seguridad en los datos. Múltiples copias de un dato incrementa la posibilidad de que alguna de ellas sea accedida sin autorización.

  • Inconsistencia de datos. Existe cuando diferentes valores de un dato se encuentran en diferentes lugares a la vez. Esto ocasionará que los resultados obtenidos sean poco fiables. La inconsistencia de datos también se le conoce como carencia de integridad de datos.

  • Anomalías en los datos. Ocurre cuando un cambio de un dato no se lleva a cabo en todos los lugares donde la redundancia de datos existe. La anomalía en los datos puede ocurrir al actualizar, insertar o borrar un dato.

Funciones de los SGBD

Un SGBD realiza varias funciones importantes que garantizan la integridad y consistencia de los datos. Cada una de esas funciones se describen a continuación:

  • Administración de un diccionario de datos. El SGBD almacena definiciones de los datos y sus relaciones (metadatos) en un diccionario de datos. Cualquier cambio que se haga a la estructura de la base de datos se almacena en el diccionario de datos, de manera que el SGBD se encarga de gestionar los cambios para que las aplicaciones y usuarios no se percaten de la ocurrencia de dicho cambio.

  • Administración del Almacenamiento de Datos. El SGBD crea y administra estructuras complejas requeridas para el almacenamiento de los datos, liberando al usuario de esta dificil tarea. Un SGBD proporciona almacenamiento de datos, definiciones, reglas de validación de datos, código procedural, estructuras para manejo de audio y video, etc. La administración de datos también es importante para la optimización del rendimiento de la base de datos, la cual determina la eficiencia en términos de almacenamiento y velocidad de acceso.

  • Presentación y administración de los datos. El SGBD transforma los datos ingresados de acuerdo a la estructura de datos requerida. Esta característica permite que el SGBD almacene los datos en un formato físico, y lo presente al usuario en un formato lógico.

  • Administración de la seguridad. El SGBD crea un sistema de seguridad que refuerza la seguridad y privacidad de los usuarios. La seguridad determina los usuarios que tienen acceso al sistema, los datos a los que tienen acceso y las operaciones que se les permite realizar (lectura, inserción, supresión o modificación).

  • Control de acceso multiusuario. Para proporcionar consistencia e integridad de datos, los SGBD utilizan algoritmos sofisticados que aseguran que múltiples usuarios tengan acceso a la base de datos sin comprometer la integridad de la base.

  • Administración de Respaldo y Recuperación. El SGBD proporciona respaldo y recuperación de datos para la seguridad e integridad de los datos. La administración de recuperación se encarga de reparar la base de datos luego de una falla, tal como un disco duro defectuoso o un apagón. Esta característica es crucial para mantener la integridad de los datos.

  • Administración de la integridad de datos. El SGBD promueve y refuerza reglas de integridad, minimizando la redundancia y maximizando la consistencia. Las relaciones de los datos almacenadas en el diccionario de datos se utiliza para reforzar la integridad de los datos.

  • Lenguajes de acceso a la base e interfaces de programación de aplicaciones. El SGBD proporciona acceso a los datos a través de un lenguaje de consulta. Un lenguaje de consulta es un lenguaje no procedural que permite que el usuario solicite lo que se debe hacer sin tener que especificar como se debe hacer. SQL (Structured Query Languaje) es un lenguaje estándar de consulta y acceso a datos soportado por la mayoría de SGBD.

  • Interfaces de comunicación a la Base de Datos. Los SGBD actuales permiten solicitudes de los usuarios a través de múltiples ambientes de red. Gracias a esto, se puede tener acceso al SGBD desde un navegador web, aplicaciones de escritorio o moviles y prácticamente cualquier dispositivo en Internet.

Componentes de las Bases de Datos

El término sistema de bases de datos se refiere a una organización de componentes que definen y regulan la recolección, almacenamiento, administración y uso de datos dentro del entorno de una base de datos.

Los componentes de una base de datos son:

  • Hardware. Se refiere a todos los dispositivos físicos del sistema. Por ejemplo: computadoras, discos duros, impresoras, dispositivos de red, etc.

  • Software.

    • Sistema Operativo. Administra los componentes de hardware y hace posible la ejecución de programas en una computadora. Ejemplos: Windows, macOS, Linux, UNIX, BSD, OS2, AIX, Solaris, etc.

    • SGBD. Sistema de software que administra la base de datos. Ejemplos: Microsoft SQL Server, Oracle MySQL (antes Sun SQL), MariaDB (Open Source), DB2 de IBM, PostgreSQL (Open Source), etc.

    • Aplicaciones y Utilidades. Programas que permiten el acceso y la manipulación de datos del SGBD. Sus funciones pueden ser desde consulta de los datos para generar reportes o hasta para la administración y control del SGBD.

  • Personas. Todos los usuarios de la base de datos. Pueden ser:

    • Administradores del sistema. Supervisan el funcionamiento del sistema.

    • Administradores de la Base de Datos. Se les conoce como DBA y se encargan de que la base de datos funcione correctamente.

    • Diseñadores de la base de datos. Se les conoce como arquitectos de la base de datos. Una base de datos bien diseñada permitirá tener un desempeño óptimo de la misma.

    • Programadores y analístas de sistema. Diseñan y crean las aplicaciones que interactúan con el sistema de base de datos.

    • Usuario final. Conjunto de personas que utilizan las aplicaciones para tener acceso a la base de datos. Utilizan la información para su trabajo, no tienen necesariamente conocimiento de la estructura de la base ni conocimientos técnicos avanzados.

  • Procedimientos. Instrucciones y reglas que rigen el diseño y uso de un sistema de base de datos.

  • Datos. El término dato se refiere al conjunto de todos los hechos almacenados en la base de datos.

Porque una Hoja de Cálculo no es una Base de Datos

Una hoja de cálculo permite la creación de múltiples tablas, pero no soporta la función de auto documentación a través de los metadatos, aplicación de tipos de datos o dominios para asegurar la consistencia de datos en una columna, relación entre tablas, restricciones para asegurar la consistencia de los datos entre tablas relacionadas. Muchos usuarios carecen del conocimiento necesario para distinguir las limitaciones de las hojas de cálculo para estas tareas.

Modelo de Datos

El diseño de una base de datos se centra en como se utilizará la estructura de la base de datos para almacenar los datos que introduzca el usuario. El modelado de dato es el primer paso para el diseño, y se refiere al proceso de crear un modelo específico de datos para un determinado problema.

Un modelo de datos es una representación relativamente simple, usualmente de forma gráfica, de estructuras de datos complejas del mundo real. El modelo de datos representa estructuras y sus características, asi como relaciones, restricciones, transformaciones -entre otros- con el propósito de representar un el dominio de un problema.

El modelado de datos es un proceso iterativo y progresivo. El proceso se inicia con un entendimiento básico del problema, conforme el entendimiento se incrementa, el nivel de detalle del modelo de datos también.

Elementos Básicos del Modelo de Datos

Los bloques básicos de construcción de un modelo de datos son: entidades, atributos, relaciones y restricciones.

  • Entidad. Culaquier cosa acerca de la cual se alamacenan datos. Una entidad representa un tipode objeto del mundo real. FDado que una entidad es representa un tipo particular de objeto las entidades son distinguibles, esto significa que cada ocurrencia de la entidad es única.

  • Atributo. Un atributo es una característica de una entidad. Un atributo es equivalente a un campo en un sistema de archivos.

  • Relación. Describe una asociación entre entidades. Los modelos de datos utilizan tres tipos de relaciones: uno a muchos (1:M), muchos a muchos (M:N) y uno a uno (1:1).

    • Uno a muchos (1:M). Un pintor (1) hace muchas pinturas (M), cada pintura es hecha por solo un pintor. La relación "PINTOR hace PINTURA" es 1:M.

    • Muchos a muchos (M:M). Un estudiante aprende varias competencias (M), cada competencia puede ser aprendida por varios estudiantes (N). La relación "ESTUDIANTE aprende COMPETENCIA" es M:N.

    • Uno a uno (1:1). Una empresa puede requerir que cada una de sus tiendas sea administrada por un solo empleado (gerente), un gerente administra una sola tienda. La relación "EMPLEADO administra TIENDA" es 1:1.

  • Restricción. Una restricción es una condición que se establece en los datos. Por ejemplo:

    • Una calificación debe estar entre 0.0 y 10.0.

    • El salario debe estar entre 6,000 y 50,000.

    • Cada clase debe tener solo un profesor.

Modelo Jerárquico

Fue desarrollado en la década 1960 para administrar grandes cantidades de datos para proyectos de fabricación, tales como el cohete Apollo en 1969.

Su estructura lógica básica se representa como un árbol invertido. En esta jerarquía, un nivel superior se percibe como padre del segmento que se encuentra directamente debajo, que se le llama hijo. El modelo jerárquico representa un conjunto de relaciones uno a muchos (1:M) entre un padre y sus hijos.

Modelo Relacional

Fue introducido en 1970 por E.F. Codd (de IBM) en su artículo "A Relational Model of Data for Large Shared Databanks" (Communications of the ACM, June 1970, pp. 377−387). El modelo representa un gran avance para los usuarios y diseñadores de las bases de datos.

El modelo relacional se fundamenta en el concepto matemático llamado "relación" (en la práctica se le llama tabla). Cada elemento en una relación se le llama tupla. Cada característica de una relación se le llama atributo. El modelo relacional también describe la manipulación de un conjunto de datos basado en conceptos matemáticos.

Modelo Entidad Relación

Introducido por Peter Chen en 1976, el modelo ER permite representar gráficamente las entidades y sus relaciones en la estructura de la base de datos.

El modelo se basa en los conceptos entidad y relación. Una entidad se representa con un rectángulo y en su interior su nombre. Las relaciones describen una asociación entre datos. La mayoría de las relaciones describen asociaciones entre dos entidades.

Modelo Orientado a Objetos (OO)

En el Modelo de Datos Orientado a Objetos (OODM), los datos y relaciones se almacenan en una sola estructura llamada objeto.

El modelo de datos OO tiene los siguientes componentes:

  • Un objeto es una abstracción de una entidad del mundo real. Un objeto representa solo una ocurrencia de una entidad.

  • Los atributos describen las propiedades de un objeto.

  • Los objetos que comparten características similares se agrupan en clases.

  • Las clases se organizan en jerarquía de clases.

  • La herencia es la capacidad de un objeto de recibir los atributos y métodos de las clases jerarquicamente superiores.

Modelo Objeto/Relacional y XML

Debido a la demanda para representar datos más complejos, se creó el Dodelo de Datos Relacional Extendido (MDRE). Este modelo (MDRE) añade muchas de las características del modelo Orientado a Obetos en una estructura relacional más simple.

El MDRE originó una nueva generación de bases de datos relacionales que soportan capacidades Orientadas a Objetos tales como: objetos, tipos de datos extendidos basados en clases y herencia.

Debido a una creciente demanda de objetos complejos en un ambiente global gracias al Internet, surgió el Lenguaje de Marcado Extensible (XML) como un estándar para hacer más eficiente el intercambio de datos estructurados, semiestructurados y no estructurados. Debido a que las organizaciones se percataron de la necesidad de manejar una cada vez mayor cantidad de datos no estructurados (documentos, páginas we, correo electrónico, hojas de cálculo, etc.), surgieron entonces las bases de datos XML. Por esta razón, es que el Modelo Objeto/Relacional añadió soporte para XML.
